## 2026-07-18 — Modul 4.14 „Gesundheit/Schädlinge" live (v1.13.0)

CH-Krankheits-Katalog + Diagnose-Journal je Volk + Meldepflicht-Hinweis. Brainstorming → Spec v2 → adversariales Review (4 Lupen, 21 Funde/18 eingearbeitet) → Plan → subagent-getriebene Umsetzung + finaler Code-Review. 96/96 Tests, live. Startet ab Volk 1 (Übernahme 19.07.).

- **D-32 · Bestandeskontroll-Spur + Katalog als Dart-const (national, kanton-neutral):** `gesundheitsereignisse` je Volk mit **Soft-Delete/Storno**, volk-FK **`ON DELETE RESTRICT`**, keine DELETE-Policy — **kein RPC, kein Immutable-Trigger** (proportional; 4.14 ist **nicht** das federal TAMV-Pflichtjournal wie 4.5; die Sperrbezirk-/Fristen-Engine ist 4.23). Das Review bestätigte diese Grenze. Der **Katalog** (17 CH-Krankheiten, `rechtskategorie` zu_bekaempfen/zu_ueberwachen/nicht/neobiota) ist eine **Dart-Fachkonstante** (wie `wirkstoff.dart`), kein DB-Seed. **Review-Kritik M1:** „Bieneninspektor GR" war ein Arosa-Hardcode in universellen Fachdaten → **kanton-neutral** formuliert + **nationale BGD-Hotline 0800 274 274**; der konkrete Inspektor-Kontakt kommt via 4.23/F4.
- **D-33 · Meldepflicht = Hinweis, Rechtskategorie = Bundesrecht (national):** Bei `zu_bekaempfen` (AFB/EFB ab Verdacht) roter Banner (Inspektor + Volk geschlossen halten), `neobiota` (Vespa) → asiatischehornisse.ch, alle mit **Rechtsauskunft-Disclaimer** („ohne Gewähr, verbindlich = Fachstelle/BLV"). Kein Zwang, keine Fristen-Engine (4.23). **Review-Korrektur M8:** die TSV-`rechtskategorie` ist **national identisch** (nicht kantonal) → bleibt dauerhaft Dart-const; kanton-/tenant-spezifisch sind nur Inspektor-Kontakt + Sperr-Radien. **M2 (einzige DB-Invariante ohne RPC):** CHECK `status='gemeldet' ⇒ gemeldet_am NOT NULL` (+ Zukunfts-Guard). **M3:** Katalog↔DB-CHECK-Paritätstest (Drift-Schutz).
- **D-34 · 4.3 ↔ 4.14 komplementär:** 4.3-`auffaelligkeiten` = Schnellflag, 4.14 = formale Diagnose. Krankheitsscharfer **Nudge** (je Flag der letzten Durchsicht ohne aktives Ereignis derselben Krankheit → „als Diagnose erfassen", `faulbrut_verdacht`→afb). **M4:** Vergiftung als eigener Katalog-Eintrag mit **eigenem Meldeweg** (BGD 0800 274 274 / Agroscope, Proben vor Regen — nicht der Inspektor).
- **Gotcha (App):** (1) Krankheits-Enum lebt zweimal (Dart-Katalog + DB-CHECK) → Paritätstest `krankheitKeys == _dbCheckKeys` (Task 3) + Checklist „neuer Key ⇒ paired G-Migration". (2) Finaler Review: `?k=`-URL-Param muss gegen den Katalog validiert werden (sonst Dropdown-`initialValue`-Crash); `aktiveMeldepflichtProvider` muss **`istMeldepflichtig`** (zu_bekaempfen **+** neobiota) filtern, sonst zeigt ein Vespa-Ereignis keinen Banner. (3) Fotos: privater Bucket `health-photos` (D02-Muster), `FotoSpeicher(client, 'health-photos')`, `gruppeId=volkId` (D-19).

## 2026-07-18 — Modul 4.6 „Fütterung" live (v1.12.0)

Fütterungs-Log (Bio-Nachweis) + Winterfutter-Ziel-Fortschritt + Material-Kopplung. Zwilling von 4.5 auf Bio-Nachweis-Niveau. Brainstorming → Spec v2 → adversariales Review (4 Lupen, 20 Funde/13 eingearbeitet) → Plan → subagent-getriebene Umsetzung + finaler Code-Review. 86/86 Tests, live.

- **D-29 · Bio-Nachweis-Niveau statt 4.5-Härte (proportional):** Fütterung ist ein **selbst-zertifizierter Bio-Nachweis** (Knospe), **kein** federal TAMV-Pflichtjournal → bewusst **Soft-Delete + Storno**, **kein Immutable-Trigger**, Edit DB-erlaubt. Das Review bestätigte diesen Leichtbau; die Skeptiker verwarfen die Forderungen nach 4.5-Immutable-Maschinerie als Over-Engineering (App-Immutabilität schafft gegen den Dateneigner keine Non-Repudiation; `updated_by` trägt das WER). **ABER (Review-Kritik M1):** volk-FK **`ON DELETE RESTRICT`** (nicht CASCADE) — sonst hätte ein Volk-Hard-Delete via `voelker_del_writer`-Policy den Log spurlos mitgerissen und die in §2/§3 zugesicherte Audit-Spur ausgehebelt. `RESTRICT` ist ein Ein-Wort-Constraint, nicht Teil der ausgeschlossenen schweren Maschinerie. Ich hatte „RESTRICT/Immutable" fälschlich als ein Ausschlusspaket gebündelt.
- **D-30 · RPC `fuetterung_erfassen` als einziger Schreibpfad** (wie 4.5): `distinct` Völker → je 1 Zeile, Lager-Abbuchung `menge_pro_volk_kg × ROW_COUNT` (nie `array_length`), `betrieb_id` explizit. **Enums (`zweck`/`futterart`) IN der RPC geprüft** → BA040 statt rohem 23514 (Review M9). Keine INSERT-Policy (Insert RPC-only), keine DELETE-Policy (Soft-Delete). BA040 Pflichtfeld/Enum, BA041 Völker, BA042 Material-Tenancy.
- **D-31 · Winterfutter-Ziel + Saison-Σ:** `winterfutter_ziel_kg` in `betriebs_einstellungen` (F4-Parameter, Default 22, CHECK > 0), mandantenfähig. **Nur `auffuetterung` zählt** fürs Ziel (Reiz/Not nicht). **Saison-Anker in `winterfutterKg` gekapselt (Review M2):** `Monat < 7 → Vorjahr` — sonst hätte `now().year` den Balken Jan–Juni auf 0 genullt (mitten in der Überwinterung), weil die Auffütter-Saison über den Jahreswechsel läuft. Fenster-Start `kAuffuetterSaisonStartMonat = 7` (Nordhalbkugel-Fachdefault, F4 später).
- **Gotcha (App):** (1) `menge_pro_volk_kg` = **Produktmasse je Volk** (Review M3 — Spalte/Label mit „pro Volk", sonst ab dem 2. Volk vervielfacht die RPC-`× v_n`-Abbuchung; M5 — Balken labelt „Produktgewicht ≠ Vorrat"). (2) Finaler Review: Balken-`(prozent*100).floor()` statt `round()` (sonst „100 %" neben amber/„nicht erreicht" bei 21.9/22); **Bio-Schalter Default `false` (fail-safe)** — `true` hätte bei vergessenem Umlegen einen stillen `bio_zertifiziert=true`-Falsch-Positiv im Bio-Nachweis erzeugt. (3) `betriebsEinstellungenProvider` steht schon seit 4.2 in `_datenNeuLaden` (der Balken liest daraus das Ziel); nur `fuetterungenFuerVolkProvider` neu ergänzt.

## 2026-07-18 — Modul 4.5 „Behandlungen (Varroa/Gesundheit)" live (v1.11.0)

TAMV-Behandlungsjournal (amtliche, revisionssichere Pflichtdaten) + Varroa-Milbendiagnose + methodenbewusstes Cockpit. Brainstorming → Spec v2 → adversariales Multi-Agent-Review (6 Lupen, 41 Funde/33 eingearbeitet) → Plan → subagent-getriebene Umsetzung + finaler Code-Review. 72/72 Tests, live.

- **D-24 · Amtliche Nicht-Löschbarkeit DB-erzwungen (Review-Kritik K1):** Der FK `behandlungen.(betrieb_id,volk_id) → voelker` ist **`ON DELETE RESTRICT`** (nicht CASCADE). *Warum:* die generische `voelker_del_writer`-Policy (A09) hätte einen Volk-Hard-Delete erlaubt, dessen CASCADE das gesamte TAMV-Journal (auch stornierte Einträge) mitgerissen und die 3-Jahres-Aufbewahrung (TAMV Art. 29) **jetzt** gebrochen hätte. Ergänzt: **keine INSERT-Policy** (Insert nur via RPC), **keine DELETE-Policy**, BEFORE-UPDATE-Trigger `behandlungen_schutz` (Kernfelder unveränderlich, Einweg-Storno `false→true`, server-seitiges `storno_am`, BA034). Korrektur = **Storno + Neueintrag** (Buchungs-Storno-Muster). Ab dem ERSTEN Eintrag scharf — nicht auf 4.23 vertagt (retroaktiv nicht heilbar).
- **D-25 · RPC `behandlung_erfassen` als einziger Schreibpfad:** Sammelbehandlung nimmt `volk_ids[]`, **dedupliziert (`distinct`)** → je eine Zeile pro Volk (TAMV-korrekt), bucht das Lager aus der **real eingefügten Zeilenzahl (`ROW_COUNT`)** ab (nie aus `array_length` → kein Doppelabzug bei Duplikaten), setzt `betrieb_id` **explizit** aus den Völkern (nicht über den JWT-Default → sonst Entkopplung Prüf-/Schreibziel bei Mehrfach-Mitgliedschaft, D-12-Lehre). Pflichtfelder konditional hart: BA030 (Kernfeld), BA031 (Völker), BA032 (Material-Tenancy), BA033 (Dosierung bei Chemie). Biotechnik/Wärme brauchen weder Präparat noch Menge.
- **D-26 · Zwei getrennte Tabellen + `material_id` `SET NULL (material_id)`:** Diagnose (`varroa_kontrollen`, normale CRUD, CASCADE) vs. amtliches Journal (`behandlungen`, RESTRICT/revisionssicher). Die Material-Komposit-FK nutzt die **spaltenqualifizierte** Form `on delete set null (material_id)` (unqualifiziert hätte PG auch `betrieb_id` NOT NULL genullt → Material unlöschbar) — konsistent mit D-15/C04.
- **D-27 · Methodenbewusste Varroa-Ampel + Thymol = bio-konform (Fachkorrektur aus Review):** Zwei Skalen — Gemüll → Milben/Tag (saisonale Schwellen Recherche 15 §4, Nov–Apr = kein Richtwert), Puderzucker/Auswaschung → Befall-% — statt einer. **Thymol/Thymovar zählt als bio-konform** (Recherche 15 §5/§7.5); die 5 mg/kg sind ein *Wachs*-Rückstandsgrenzwert, keine Aussage zur Behandlung → v1 stufte es fälschlich als „grenzwertig" ein (hätte Cry-Wolf auf der alpinen Regelbehandlung erzeugt). Nur `sonstige` löst die Bio-Warnung aus. Schwellen/Höhen-Offset sind universelle Fachdefaults (F4-konfigurierbar), kein Arosa-Hardcode; Interim: Höhen-Caveat im Cockpit.
- **D-28 · Errcode-Block BA030–039 = Modul 4.5** (BA030 Pflichtfeld, BA031 Völker, BA032 Material-Tenancy, BA033 Dosierung, BA034 Revisionssicherheit/Immutable-Trigger).
- **Gotcha (App):** (1) Sammelbehandlung invalidiert **jede** beteiligte Volk-Family (`for id in volkIds: invalidate(behandlungenFuerVolkProvider(id))`) **plus `materialListProvider`** — nur `invalidateSelf()` ließe die übrigen Völker stale (D-18/D-23-Gotcha, intra-Mandant). (2) `behandlungen` hat **keine INSERT-Policy** → jeder Journaleintrag MUSS über die RPC (der Fake-Gateway bildet distinct/Validierung nach). (3) Finaler Code-Review fand: `behandlung_section` maskierte einen Journal-Ladefehler als Endlos-Spinner (nur die Kontrollen-Seite war abgedeckt) → beide AsyncError-Positionen zeigen jetzt den Fehler.
- **Test-Hook (DB):** RPC-DO-Tests im Admin-Kontext (`execute_sql`) über `set_config('app.current_user_id', <uid>, true)` — `private.current_app_user()` liest das GUC vor `auth.uid()`, so wird `kann_schreiben` für den Owner true.

## 2026-07-17 — Modul 4.3 „Durchsicht/Stockkarte" live (v1.10.0)

Digitale Stockkarte je Volk (fünf Kernfragen W-B-F-P-G). Spec→Plan→Umsetzung, subagent-getrieben, 4-fach reviewt (Design-Review 25 Findings/14 eingearbeitet). 53/53 Tests, live.

- **D-19 · Foto-Datenschutz:** Durchsichts-Fotos (Brutbild/Krankheiten = Gesundheitsdaten) liegen im **privaten** Bucket `inspection-photos` (anders als die public Bau-Fotos), `foto_urls` speichert **Pfade**, Anzeige via **Signed-URL**. `loeschen` entfernt die Storage-Objekte aktiv (Löschpflicht); der Volk-Hard-Delete-Pfad (CASCADE) hinterlässt best-effort Orphans → 4.25 räumt serverseitig nach.
- **D-20 · Record-only:** Eine Durchsicht ist reine Beobachtung und mutiert `voelker` **nicht** (kein Auto-Umweiseln/Statusflip). „weisellos"/„faulbrut_verdacht" wird in der Timeline sichtbar; Handeln läuft über 4.2/4.5/4.14.
- **D-21 · Kein `tasks`-Modul jetzt:** statt „Folge-Aufgabe erzeugt `tasks`" nur ein Empfehlungsdatum `naechste_durchsicht_am` am Eintrag. Voice/Offline/konfigurierbare Feldsätze vertagt.
- **D-22 · `auffaelligkeiten` als `text[]` mit DB-CHECK** (`<@`-Whitelist, 9 Werte inkl. der meldepflichtigen `faulbrut_verdacht`/`sauerbrut_verdacht`) — harte Integrität in der DB, freundliche Filterung in der App.
- **D-23 · „letzte Durchsicht je Volk" via View** `v_letzte_durchsichten` (`security_invoker`, `distinct on`) statt N Family-Loads — PostgREST kann kein `distinct on`.
- **Notiz:** Foto-Storage-Pfad ist `<betrieb_id>/<volk_id>/…` (nicht `<inspection_id>`), weil eine neue Durchsicht beim Foto-Upload noch keine id hat — RLS greift auf das erste Pfadsegment (`betrieb_id`), also mandanten-sicher.
- **Gotcha (App):** neue Family-Provider (`durchsichtenFuerVolkProvider`, `letzteDurchsichtenProvider`) in `AuthController._datenNeuLaden()` invalidieren (Fremd-Cache); vollflächige Formular-Routen brauchen einen eigenen Rollen-Guard (`viewer` → „Nur Lesezugriff"), da per URL direkt erreichbar (anders als 4.2-Modals).

## 2026-07-17 — Modul 4.2 „Völker & Standorte" live (v1.9.0)

Erstes Fachmodul auf dem Auth-Fundament. Spec→Plan→Umsetzung (Fable 5), subagent-getrieben, 3-fach reviewt (adversariales Design-Review: 43 Findings/36 eingearbeitet; + Spec- und Code-Qualitäts-Review je Milestone; + holistischer End-Review). 45/45 Tests, live deployed.

Arosa = Daten (Ops-Seed), kein Code. *Gotcha:* KEIN `default aktive_betrieb_id()` — der Gründer hat beim `betrieb_gruenden` noch keinen JWT-Claim → NULL → PK-Verletzung; die RPC übergibt `betrieb_id` **explizit** + Backfill für Bestandsbetriebe.

Bestandslücke `scales.volk_id` mitgehärtet.
- **D-16 · Errcode-Registry:** BA001–BA013 = Auth-Fundament, **BA020+ = Modul 4.2** (BA010/BA012 waren live doppelt belegt). Je Modul ein neuer Zehnerblock; Prüfschritt `grep BA0` vor neuen RPCs.
- **D-17 · Königin↔Volk 1:1 als DB-Garantie** (partieller Unique-Index `voelker(koenigin_id) where … not null`) — BA022 galt vorher nur im RPC, der CRUD-Pfad umging sie.
- **D-18 · Navigation:** „Völker" wird Haupttab (Drehscheibe); „Recherche"/„Entscheidungen" ins „Mehr"-Menü.
- **Gotcha (App):** (1) Neue Daten-Provider MÜSSEN in `AuthController._datenNeuLaden()` invalidiert werden — sonst Fremd-Mandanten-Cache nach Login-Wechsel. (2) Formular-Dropdowns: Stammdaten vor dem Öffnen `await …provider.future` (AsyncNotifier liefert beim ersten `read` noch `AsyncLoading` → leere Dropdowns). (3) PostgREST-Relation-Select nutzt die FK-**Constraint-Namen** (`voelker_koenigin_fk`/`voelker_standort_fk`).

## 2026-07-16 — Auth-Fundament live (Rollout & Cutover)

- **Cutover vollzogen:** Das Fundament ist scharf. `anon` hat **keinen** Zugriff mehr (Policies + Table-Grants entzogen); ab jetzt gilt echte Mandanten-Isolation über die `authenticated`-RLS. Owner = Daniel (`dani.proyer@gmail.com`), Betrieb **„Imkerei-Projekt Arosa"** (`1c84d5dd-…`). App **v1.8.1** live.
- **Erkenntnis (Gotcha, wichtig):** Der Custom-Access-Token-Hook setzt Custom-Claims in die **JWT-Claims**, NICHT in `auth.users.raw_app_meta_data`. Client-seitig MUSS man den Claim aus dem **dekodierten Access-Token** lesen (`jwtPayload()`), nicht aus `session.user.appMetadata`. Sonst „ohneBetrieb"-Dauerschleife. → Regressionstest vorhanden.
- **Erkenntnis:** Ein `betrieb_id`-einfrierender UPDATE-Trigger (`set_row_actor`) und ein Backfill (`UPDATE … WHERE betrieb_id IS NULL`) beißen sich. Lösung: nur einfrieren, wenn bereits gesetzt (`coalesce(old, new)`) — Migration A13.
- **Config-Lehre:** Supabase-**Site URL** ≠ Redirect-Allowlist. Der Bestätigungslink nutzt die Site URL; bleibt sie auf `localhost:3000`, geht der Link ins Leere (die E-Mail wird server-seitig trotzdem bestätigt).
- **Offen/vorgemerkt:** Leaked-Password-Protection aktivieren; Lorena einladen wenn bereit.
